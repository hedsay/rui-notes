# 创建线程方式

  1. Thread方式
public static void main(String[] args) {
    new Thread(()->{
        System.out.println("线程开始");
        try {
            Thread.sleep(1000);
        } catch (InterruptedException e) {
            e.printStackTrace();
        }
        System.out.println("线程结束");
    }).start();
}

  1. Runnable配合Thread

线程与任务分开，更容易与线程池等高级API配合，更加灵活

public static void main(String[] args) {
    Runnable runnable = new Runnable() {
        @Override
        public void run() {
            System.out.println("线程开始");
            try {
                Thread.sleep(1000);
            } catch (InterruptedException e) {
                e.printStackTrace();
            }
            System.out.println("线程结束");
        }
    };
    Thread thread = new Thread(runnable);
    thread.start();
}

  1. FutureTask

接受Callable类型的参数，处理有返回结果的情况

public static void main(String[] args) throws ExecutionException, InterruptedException {
    FutureTask<Integer> task = new FutureTask<>(() -> {
        System.out.println("线程运行后");
        return 100;
    });
    new Thread(task).start();
    Integer res = task.get();
    System.out.println("线程得到结果：" +res);
}

# 如何查看线程/进程

windows:

  • tasklist 查看
  • taskkill /pid 杀死

linux：

  • ps -fe 查看所有
  • ps -fT -p 查看某个
  • kill 杀死
  • top 按 H 切换是否显示线程
  • top -H -p 查看某个进程（PID）的所有线程

Java:

  • jps 查看所有 Java 进程
  • jstack 查看某个 Java 进程（PID）的所有线程状态
  • jconsole 查看某个 Java 进程中线程的运行情况（图形界面）

# 进程与线程比较

  1. 进程是资源分配的单位，线程是CPU调度的单位；
  2. 线程创建比进程快，进程创建过程中还需要资源管理信息，如内存管理信息，文件管理信息，而线程不涉及这些管理信息，只是共享它们；
  3. 同一个进程内的线程比进程切换快，线程虚拟内存共享，意味着同一个进程的线程具有同一个页表，切换线程时不需要切换页表。进程切换切换页表开销很大；
  4. Java中，启动main函数就启动了一个JVM进程，main函数所在线程就是进程中的一个线程；
  5. 同类多个线程共享进程的堆和方法区，每个线程独享自己的程序计数器，虚拟机栈吗，本地方法栈；

# 程序计数器&虚拟机栈&本地方法栈为什么私有？

程序计数器私有主要是为了线程切换后能恢复到正确的执行位置；

虚拟机栈：每个 Java 方法在执行之前会创建一个栈帧用于存储局部变量表、操作数栈、常量池引用等信息；

本地方法栈：区别于虚拟机栈，本地方法栈是虚拟机使用到的 Native 方法，会创建一系列信息；

虚拟机栈&本地方法栈线程私有，为了保证线程中的局部变量不被别的线程访问到；

# 堆与方法区为什么线程共享？

堆是进程中最大的一块内存，主要用于存放新创建的对象 (几乎所有对象都在这里分配内存)；

方法区主要用于存放已被加载的类信息、常量、静态变量、即时编译器编译后的代码等数据；

# 线程上下文切换时机

  • 线程的 cpu 时间片用完
  • 有更高优先级的线程需要运行
  • 线程自己调用了 sleep、yield、wait、join、park、synchronized、lock 等方法
  • 垃圾回收

# 线程生命周期和状态

  • 线程创建后未被调用，处于初始状态；
  • 调用start()方法后，处于可运行状态；
  • 可运行状态的线程获得了 CPU 时间片，处于运行态；
  • 当线程执行 wait()方法之后，进入等待状态；
  • 等待状态加了超时限制，进入超时等待状态，结束后返回可运行态；
  • 线程进入synchronized或wait后重新进入synchronzied方法，但锁被其它线程占有，进入阻塞态；
  • 线程执行完run()方法之后，进入终止态；

# 两阶段终止

isInterrupted:不会清除打断标记

interrupted:会清除打断标记

更优雅的终止线程，流程图：

案例代码：

public class TwoPhaseTermination {
    private Thread monitor;
    //启动监控线程
    public void start(){
        monitor = new Thread(()->{
            while(true){
                Thread current = Thread.currentThread();
                //设置完打断标记后，再进入循环
                if(current.isInterrupted()){
                    log.debug("料理后事");
                    break;
                }
                try {
                    //此处打断并不会设置打断标记,进入捕获异常
                    Thread.sleep(1000);
                    log.debug("执行监控记录");
                } catch (InterruptedException e) {
                    e.printStackTrace();
                    //设置打断标记，如果没有会一直while循环
                    current.interrupt();
                }
            }
        });
        monitor.start();
    }
    //停止监控线程
    public void stop(){
        monitor.interrupt();
    }
}

# park打断

  • 属于LockSupport中方法，不会清空打断状态

  • 一旦打断后，再也不能park了，这个特性可以跟interrupted配合使用，实现再次打断

    案例代码:

    private static void test() throws InterruptedException {
    Thread t = new Thread(() -> {
        log.debug("park...");
        LockSupport.park();//无interrupt，下面程序不会运行
        log.debug("unpark...");
        log.debug("打断状态：{}", Thread.currentThread().interrupted());

        //isInterrupted未清楚打断标记，true,还会往下运行
        LockSupport.park();
        log.debug("unpack...");
    }, "t1");
    t.start();

    sleep(1);
    t.interrupt();
}

# sleep()&wait()

  • sleep()没释放锁，wait()释放锁
  • wait()方法需要别的线程调用同一个对象上的notify()方法才能苏醒，sleep()在方法执行完，会自动苏醒；
  • sleep()是Thread类静态本地方法，wait()是Object类本地方法；

wait()为什么定义在Object中？

wait()会释放当前线程占有的对象锁，既然释放对象锁并进入waiting状态，操作就应该是对象非当前线程；

# 可以直接调用Thread类的run方法吗？

直接执行run方法，会把run方法当成一个main线程下的普通方法去执行，并不会在某个线程中执行它，所以这并不是多线程工作。
