# JVM，JDK，JRE三者的之间的联系？

JVM：运行Java字节码的虚拟机，Java语言能够"一次编译，随处运行"就是因为使用相同的字节码，JVM可以在不同系统上给出相同结果；

JRE：Java运行时的环境，包含程序运行环境和类库；

JDK：包含了JRE，还包含可用于Java程序开发和调试的 如javac,javap,jconsole等工具；

# 基本数据类型&包装类

基本类型 字节 包装类
byte 1 Byte
short 2 Short
int 4 Integer
long 8 Long
char 2 Float
float 4 Double
double 8 Character
boolean Boolean

存储方式：基本数据类型的局部变量存放在虚拟机栈中的局部变量表，成员变量(无static修饰)存放在堆中；包装类型几乎所有实例都在堆中；

比较方式：包装类型equals()；基本类型==

不是所有对象实例都在堆中，JIT优化会进行逃逸分析，若发现某一个对象没有逃逸到方法外部，可以通过标量替换实现栈上分配；

基本数据类型的成员变量建议static，避免存放堆中；

装/拆箱：

Integer i = 10 等价于 Integer i = Integer.valueOf(10);

int n = i 等价于 int n = i.intValue();

# 包装类缓存机制

缓存提升性能，Byte,Short,Integer,Long创建了[-128,127]相应类型的缓存数据，Character创建了[0,127]缓存数据，Boolean返回T/F;

例题：

Integer i1 = 40;
Integer i2 = new Integer(40);
System.out.println(i1==i2);
1
2
3

输出false,i1是缓存的对象，i2是new对象；

.equals会输出true;

# ==和equals()

==基本类型比较值，引用类型比较地址

equals只用于判断引用类型，没有重写等价于==，重写了比较对象属性；

# hashCode()

获取哈希码，获取对象在哈希表中的索引位置；

为什么要有hashCode()?

以HashSet为例，hashcode不同的对象必然不同，若相同再调用equals检查hashcode相同的对象是否真的相同，大大减少equals次数，提高执行速度；

重写equals()为什么要重写hashCode()?

equals判断相等的对象hashCode一定相等，如果重写了equals不重写hashcode，可能会导致equals判断相等的对象hashCode不相等；
